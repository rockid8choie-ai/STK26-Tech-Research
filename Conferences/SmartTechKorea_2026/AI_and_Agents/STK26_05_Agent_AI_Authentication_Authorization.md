# 강의 요약

본 강의는 에이전트 AI 시대에 필요한 인증과 권한 관리 구조를 설명하였다.

기존 인증은 사람 또는 머신의 신원을 확인하는 데 초점이 있었다. 그러나 에이전트 AI는 단순히 “누구인가”를 확인하는 것만으로는 부족하다. 에이전트가 어떤 권한으로, 어떤 목적을 위해, 어떤 단위 행동을 수행하는지까지 검증해야 한다.

강연에서는 에이전트 AI 보안의 핵심으로 Mandate, 즉 행동허가증 기반의 최소권한 모델을 제시하였다. 또한 장수명 토큰과 기존 RBAC 권한 체계가 에이전트 환경에서는 위험할 수 있으며, 사용자 의도와 실제 API 호출을 암호학적으로 강결합해야 한다고 설명하였다.

하드웨어 보안 모듈(HSM), 단기 자격증명, 제로 트러스트 아키텍처, PQC(양자내성암호) 전환 역시 에이전트 AI 시대의 핵심 보안 요소로 소개되었다.

---

# 핵심 개념

## 에이전트 AI 인증

기존 인증은 사용자가 누구인지 또는 머신이 진짜인지 확인하는 데 집중했다.

하지만 에이전트 AI 인증은 다음을 함께 확인해야 한다.

* 에이전트가 누구인가
* 어떤 사용자의 의도에 따라 동작하는가
* 어떤 권한을 위임받았는가
* 어떤 단위 행동을 수행하는가
* 그 행동이 허가 범위 안에 있는가

즉, 에이전트 인증은 신원 확인을 넘어 실행 단위의 검증까지 포함한다.

---

## 사람 인증, 머신 인증, 에이전트 인증의 차이

### 사람 인증

사용자가 시스템에 로그인하면 해당 시스템 안에서 허용된 작업을 수행할 수 있다.

문제는 로그인 이후의 세부 행동이 넓은 권한 안에서 처리될 수 있다는 점이다.

### 머신 아이덴티티

서버, 워크로드, 데이터베이스 등 특정 머신이나 서비스가 진짜인지 증명하는 방식이다.

키가 유출되면 해당 워크로드 전체가 노출될 위험이 있다.

### 에이전트 인증

에이전트가 어떤 권한으로 어떤 행동을 하는지까지 검증해야 한다.

따라서 인증보다 권한 위임과 실행 통제가 더 중요해진다.

---

# Mandate 기반 최소권한 모델

## Mandate란?

Mandate는 에이전트에게 특정 행동을 허가하는 행동허가증이다.

단순한 접근 권한이 아니라 다음 요소를 구체적으로 묶는다.

* 사용자 의도
* 수행할 행동
* 대상
* 파라미터
* 유효시간
* 허용 범위

---

## 예시

사용자 의도:

“엄마에게 500만 원을 이체해줘.”

Mandate에는 다음 정보가 포함된다.

* 행동: 이체
* 금액: 500만 원
* 수취인: 엄마
* 유효시간: 1~5분
* 허용 범위: 해당 이체 1건

이 범위를 벗어나는 행동은 차단된다.

---

## 기존 RBAC의 한계

기존 역할 기반 권한(RBAC)을 에이전트에게 그대로 부여하면 과도한 권한이 생길 수 있다.

예를 들어 에이전트에게 파일 삭제 권한을 넓게 부여하면 사용자의 의도보다 더 많은 파일을 삭제할 위험이 있다.

따라서 에이전트 환경에서는 장수명 토큰과 포괄적 권한 대신 짧은 시간 동안 특정 행동만 허용하는 방식이 필요하다.

---

# 사용자 의도와 API 호출의 강결합

에이전트 보안에서 중요한 것은 사용자의 의도와 실제 실행된 API 호출이 연결되어 있음을 증명하는 것이다.

이를 위해 강연에서는 암호학적 강결합(Strong Binding)을 강조하였다.

예를 들어 사용자가 승인한 것은 “500만 원 이체”인데, 실제 API 호출이 다른 금액이나 다른 수취인으로 바뀐다면 이를 탐지하고 차단해야 한다.

---

# HITL과 승인 구조

## HITL이란?

HITL(Human in the Loop)은 에이전트가 중요한 행동을 수행하기 전에 인간의 승인이나 개입을 요구하는 구조이다.

---

## 한계

HITL이 있다고 해서 항상 안전한 것은 아니다.

에이전트가 대량의 승인 요청을 반복적으로 생성하면 사용자가 무심코 승인할 수 있다.

따라서 승인 UX와 검증 구조도 함께 설계되어야 한다.

---

# 표준 동향

## Google AP2

Google AP2는 에이전트 결제 프로토콜로 소개되었다.

Intent Mandate, Cart Mandate, Payment Mandate와 같이 사용자 의도와 결제 실행 단계를 암호학적으로 연결하는 구조를 가진다.

---

## FIDO Alliance

FIDO Alliance에서는 에이전트 인증과 관련된 기술 워킹그룹이 구성되어 논의가 진행 중인 것으로 소개되었다.

---

## OpenID Authorization API

OpenID Authorization API 1.0은 컨트롤 플레인 안에서 정책 결정과 권한 부여를 연결하는 표준 흐름으로 언급되었다.

---

# 하드웨어 기반 보안

## 왜 HSM이 필요한가?

소프트웨어 안에 숨긴 키는 AI 에이전트가 찾아낼 수 있다.

사람이 찾기 어렵게 숨긴 키라도 에이전트는 자동화된 탐색과 추론을 통해 노출시킬 가능성이 있다.

따라서 키와 크리덴셜은 소프트웨어 영역이 아니라 하드웨어 경계 안에서 보호해야 한다.

---

## HSM의 역할

HSM(Hardware Security Module)은

* 키 보호
* 크리덴셜 발행
* 암호 연산
* 실행 로그 보관

을 하드웨어 수준에서 수행한다.

에이전트가 접근하기 어려운 마지막 보안 경계로 기능한다.

---

# PQC 전환

## 양자내성암호의 필요성

강연에서는 2028~2030년 사이 ECC와 RSA 기반 암호 체계가 취약해질 수 있다는 전망을 언급하였다.

특히 ECC는 암호화폐, SSL 인증서 등 광범위하게 사용되므로, 장기적 관점에서 PQC 전환이 필요하다고 설명하였다.

---

## 에이전트 시스템과 PQC

에이전트가 수행하는 행동과 권한 위임이 암호학적으로 보호되어야 한다면, 그 암호 기반 역시 장기적으로 안전해야 한다.

따라서 에이전트 AI 시스템은 초기 설계 단계부터 PQC 전환 가능성을 고려해야 한다.

---

# 에이전트 AI 제로 트러스트 아키텍처

에이전트 AI 환경에서는 기존 제로 트러스트 모델만으로는 충분하지 않을 수 있다.

필요한 요소는 다음과 같다.

* 컨트롤 플레인
* 정책 엔진
* PDP(Policy Decision Point)
* PAP(Policy Administration Point)
* 마이크로 세그멘테이션
* 단기 자격증명
* Mandate 기반 권한 위임
* 실행 로그와 감사 추적

---

# 핵심 인사이트

## 1. 에이전트 인증은 신원 확인보다 행동 통제가 중요하다

에이전트가 누구인지보다 더 중요한 것은 무엇을 할 수 있는지와 어떤 의도로 행동하는지이다.

---

## 2. 에이전트에게 장수명 토큰을 주는 것은 위험하다

에이전트는 빠르게 반복 실행되며 24시간 동작할 수 있기 때문에, 긴 유효기간을 가진 토큰은 오남용 위험이 크다.

---

## 3. Mandate는 에이전트 시대의 새로운 권한 단위가 될 수 있다

기존 권한이 역할 중심이었다면, 에이전트 환경에서는 특정 행동 단위의 권한 위임이 중요해진다.

---

## 4. 하드웨어 보안은 에이전트 시대의 마지막 방어선이다

AI가 소프트웨어 영역을 탐색하고 악용할 수 있다면, 키와 실행 증거는 HSM 같은 하드웨어 기반 보안 영역에서 보호해야 한다.

---

## 5. PQC는 장기 보안 설계의 필수 요소가 되고 있다

에이전트 시스템이 장기간 운영될수록 양자내성암호 전환은 선택이 아니라 필수에 가까워질 수 있다.

---

# 주요 시사점

에이전트 AI가 실제 행동을 수행하는 시스템으로 발전하면서 인증과 권한 관리의 기준도 변화하고 있다.

기존의 로그인, 토큰, 역할 기반 권한 관리만으로는 에이전트가 수행하는 세부 행동을 충분히 통제하기 어렵다.

따라서 에이전트 AI 시스템은 사용자 의도, 행동 범위, 유효시간, 실행 로그를 하나의 권한 단위로 묶는 구조가 필요하다.

또한 소프트웨어 보안만으로는 한계가 있기 때문에 HSM, 단기 자격증명, PQC, 제로 트러스트 아키텍처를 함께 고려해야 한다.

---

# 한 줄 정리

에이전트 AI 시대의 보안은 “누가 로그인했는가”가 아니라 “어떤 의도로 어떤 행동을 어디까지 허가받았는가”를 증명하는 방향으로 진화하고 있다.
